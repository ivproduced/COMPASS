## 1. Executive Summary

COMPASS is a real-time voice agent that guides security architects through FedRAMP authorization by listening to system descriptions, analyzing architecture diagrams via vision, dynamically mapping to NIST SP 800-53 Rev 5 controls, identifying compliance gaps, and generating OSCAL-formatted output — all through natural conversation powered by Gemini 2.5 Pro Live API.

This document defines the full technical architecture, covering every layer from presentation through data persistence, along with integration contracts, security model, and deployment topology.

## 3. Architecture Principles

| # | Principle | Rationale |
|---|---|---|
| AP-1 | **Voice-First, Multi-Modal** | Core interaction is real-time voice; vision and text are augmentation channels, not replacement. |
| AP-2 | **Agent Decomposition** | Each compliance domain (classification, mapping, gap analysis, OSCAL generation) is a discrete agent with a focused prompt and tool set. |
| AP-3 | **Grounded Responses Only** | Every control reference must cite a specific control ID. No hallucinated controls — use RAG retrieval before generation. |
| AP-4 | **OSCAL-Native Output** | All structured output conforms to NIST OSCAL 1.1.2 schema. No proprietary formats. |
| AP-5 | **Stateful Sessions** | Assessments persist across interruptions. An architect can resume a session hours or days later. |
| AP-6 | **Serverless-First** | Cloud Run scales to zero. No idle cost. Burst scales for concurrent sessions. |
| AP-7 | **Security by Default** | The system that assesses security must itself be secure — encrypted transit, IAM-only auth, no long-lived credentials. |
| AP-8 | **Interruptible Conversation** | The Live API supports server VAD and turn-taking. COMPASS never blocks the architect from redirecting. |

### 4.2 Layer Responsibilities

#### Presentation Layer
- Captures microphone audio (PCM 16kHz mono) and streams to backend
- Plays back audio responses from Gemini
- Renders real-time transcript, control mapping panel, gap dashboard
- Handles architecture diagram upload (drag-drop, file picker, clipboard paste)
- Displays OSCAL output with syntax highlighting and download

#### API Gateway Layer
- **WebSocket `/ws/live`**: Bidirectional audio + JSON message relay between client and Gemini Live API
- **REST `/api/sessions`**: CRUD for assessment sessions
- **REST `/api/assessments/{id}`**: Retrieve assessment state, control mappings, gap findings
- **REST `/api/oscal/{id}/{type}`**: Download generated OSCAL documents (SSP, POA&M)
- **REST `/api/diagrams`**: Upload architecture diagrams
- Authentication via Google Identity Platform (JWT validation)
- Rate limiting, request validation, CORS

#### Agent Orchestration Layer
- **Root Agent** (`compass_root`): Manages conversation flow, routes to sub-agents based on conversation phase, maintains session context
- **Classifier Agent**: Receives system description → outputs FIPS 199 classification
- **Mapper Agent**: Receives components/descriptions → outputs control mappings via RAG
- **Gap Agent**: Compares implemented controls vs. baseline → outputs gap findings with remediation
- **OSCAL Agent**: Aggregates session findings → generates validated OSCAL JSON

#### Tools Layer
- Pure functions callable by agents via ADK function calling
- Each tool has defined input/output schemas
- Tools are deterministic where possible (classification, OSCAL generation)
- RAG-powered tools use Vertex AI Vector Search (control lookup, search)

#### Data & Storage Layer
- **Firestore**: Session documents, system profiles, control assessments, gap findings
- **Cloud Storage**: OSCAL JSON files, uploaded architecture diagrams, session exports
- **Vertex AI Vector Search**: Pre-embedded 800-53 controls, FedRAMP baselines, AI overlay controls, MITRE ATLAS mappings

### 6.2 Vertex AI Vector Search Index

**Index:** `compass-controls-index`

| Document Type | Count | Fields Embedded | Chunk Strategy |
|---|---|---|---|
| NIST 800-53 Rev 5 Controls | ~1,189 | Title + Description + Supplemental Guidance | One document per control/enhancement |
| FedRAMP Moderate Baseline | 325 | Control + FedRAMP-specific parameters | One document per baselined control |
| AI Control Overlay | 36 | Full control text | One document per AI control |
| MITRE ATLAS Mappings | ~66 | Technique description + mapped controls | One document per technique |
| Implementation Patterns | ~50 | Common implementation narratives | One document per pattern |

**Embedding Model:** `text-embedding-005` (768 dimensions)  
**Distance Metric:** Cosine similarity  
**Approximate Neighbors:** 10 (default retrieval count)

## 7. Integration Architecture

### 7.1 Gemini Live API Integration

```
┌──────────┐     WebSocket      ┌───────────┐     google-genai SDK     ┌──────────────┐
│  Browser  │ ◄──────────────► │  FastAPI   │ ◄────────────────────► │ Gemini 2.5    │
│  Client   │  PCM audio +     │  Backend   │  LiveConnectConfig     │ Pro Live API  │
│           │  JSON messages    │            │  Audio + Function Call │              │
└──────────┘                    └───────────┘                         └──────────────┘
```

**Connection Flow:**

1. Client opens WebSocket to `/ws/live`
2. Backend authenticates JWT, creates/resumes Firestore session
3. Backend opens `client.aio.live.connect()` to Gemini Live API with:
   - `response_modalities=["AUDIO", "TEXT"]`
   - Voice config: `Kore` (professional, clear)
   - Function declarations for all tools
   - System instruction with COMPASS persona + compliance context
4. Two async tasks run concurrently:
   - **Receive loop**: Client audio → Gemini (`LiveClientRealtimeInput`)
   - **Send loop**: Gemini responses → Client (audio bytes + JSON transcript + tool call results)
5. On function call: backend executes tool, returns `LiveClientToolResponse`
6. On disconnect: session state saved to Firestore

**Vision Input Flow:**

1. Client uploads diagram via REST `/api/diagrams` → stored in GCS
2. Client sends JSON message over WebSocket: `{"type": "diagram", "url": "gs://..."}`
3. Backend sends image to Gemini session: `session.send(input=types.LiveClientContent(parts=[types.Part(inline_data=image_blob)]))`
4. Gemini analyzes diagram and responds with component identification + control observations

### 7.2 Vertex AI Vector Search Integration

```
┌────────────────┐     Embedding Request     ┌─────────────────┐
│  Tool Function  │ ──────────────────────► │ text-embedding-  │
│  (search_       │                          │ 005              │
│   controls)     │ ◄────────────────────── │                  │
│                 │     768-dim vector        └─────────────────┘
│                 │
│                 │     Nearest Neighbor      ┌─────────────────┐
│                 │ ──────────────────────► │ Vertex AI        │
│                 │                          │ Vector Search    │
│                 │ ◄────────────────────── │ Index            │
└────────────────┘     Top-K results         └─────────────────┘
```

**Query Flow:**

1. Agent calls `search_controls(query="web application firewall for boundary protection")`
2. Tool embeds query via `text-embedding-005`
3. Tool queries Vector Search index with embedding vector, `k=10`
4. Returns top-K control documents with similarity scores
5. Agent uses retrieved controls to ground its response

## 8. Security Architecture

### 8.1 Authentication & Authorization

| Layer | Mechanism |
|---|---|
| **Client → Backend** | Google Identity Platform (Firebase Auth) — JWT bearer tokens |
| **Backend → Gemini** | Service account with `aiplatform.user` role |
| **Backend → Firestore** | Service account with scoped Firestore access |
| **Backend → GCS** | Service account with scoped storage access |
| **Backend → Vector Search** | Service account with `aiplatform.user` role |

### 8.2 Data Protection

| Data | At Rest | In Transit |
|---|---|---|
| Audio streams | Not persisted (real-time only) | TLS 1.3 (WebSocket WSS) |
| Transcripts | Firestore (encrypted at rest by default) | TLS 1.3 |
| OSCAL outputs | GCS (AES-256 server-side encryption) | TLS 1.3 |
| Architecture diagrams | GCS (AES-256) | TLS 1.3 |
| Vector embeddings | Vertex AI (encrypted at rest) | TLS 1.3 (gRPC) |
| Session tokens | Firebase Auth (short-lived JWTs) | TLS 1.3 |

### 8.3 Network Security

- Cloud Run service with **ingress = all** (public endpoint, auth required)
- No VPC connector needed for initial deployment (serverless-to-serverless)
- Cloud Armor WAF in front of Cloud Run for production hardening
- CORS restricted to frontend domain origin

### 8.4 Secrets Management

- All API keys and service account credentials stored in **Secret Manager**
- Cloud Run pulls secrets as environment variables at deploy time
- No long-lived credentials in code or config files

## 10. Performance & Scalability

### 10.1 Latency Budgets

| Operation | Target | Notes |
|---|---|---|
| Audio round-trip (user speaks → COMPASS responds) | < 1.5s | Gemini Live API handles streaming; sub-second for short turns |

### 10.3 Concurrency Model

- Each Cloud Run instance handles **1 concurrent WebSocket session** (due to Gemini Live API connection being stateful)
- Cloud Run max-concurrency set to **1** for the backend service
- Scale-out is horizontal: new session → new instance
- REST API endpoints can share instances (separate service or higher concurrency)

## 11. Error Handling & Resilience

### 11.1 Error Categories

| Category | Handling Strategy |
|---|---|
| **Gemini Live API disconnection** | Detect WebSocket close → notify client → auto-reconnect with session state from Firestore |
| **Tool execution failure** | Return structured error to agent → agent communicates gracefully to architect ("I couldn't look up that control — let me try a different approach") |
| **Vector Search timeout** | Fallback to cached control catalog (local JSON) for deterministic lookup by control ID |
| **Firestore write failure** | Retry with exponential backoff (3 attempts) → buffer in-memory → flush on recovery |
| **OSCAL validation failure** | Return validation errors to agent → agent asks for clarification or fixes data |
| **Audio stream corruption** | Gemini handles server-side VAD; client-side: detect silence > 30s → prompt user |

### 11.2 Session Recovery

1. All session state persisted to Firestore after each significant action (classification, control mapping, gap finding)
2. On reconnect, backend loads session from Firestore
3. Gemini session re-initialized with system instruction + session context summary
4. Agent greets: "Welcome back. We were working on [system name]. We've mapped [N] controls so far. Shall we continue with [next phase]?"
